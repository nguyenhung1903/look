## Two-stage retrieval update (quality + performance)

What we changed:

- Kept safety/correctness changes from PR#19 (mutex poisoning recovery, URL encoding cleanup, translation target validation).
- Replaced full-scan expensive fuzzy scoring with a two-stage approach:
  - Stage 1: fast scoring across all candidates.
  - Stage 2: quality bonus rerank only on top-N candidates.
- Added code comments in `core/engine/src/search.rs` explaining the two-stage design and latency goal.
- Added `fuzzy_quality_bonus_prepared` in `core/matching/src/lib.rs` and tests for quality-bonus behavior.
